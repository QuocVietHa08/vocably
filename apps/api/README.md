# Vocally API

FastAPI backend for Vocally — AI-powered vocabulary recommendation engine with **LangChain**, **spaced repetition (SM-2)**, and **CEFR level detection**.

## Prerequisites

- **Python 3.12+**
- **pip** (comes with Python)
- **Supabase project** with migrations applied
- **OpenAI API key** (for LangChain AI ranking)

## Setup

```bash
# 1. Navigate to the API directory
cd apps/api

# 2. Create a virtual environment
python -m venv venv

# 3. Activate the virtual environment
source venv/bin/activate        # macOS / Linux
# venv\Scripts\activate         # Windows

# 4. Install dependencies
pip install -r requirements.txt

# 5. Configure environment variables
cp .env.example .env
# Edit .env with your actual keys (see Environment Variables below)

# 6. Run the database migrations
# Go to your Supabase Dashboard → SQL Editor → paste and run IN ORDER:
#   1) supabase/migrations/001_initial_schema.sql      (profiles, decks, flashcards, etc.)
#   2) supabase/migrations/002_recommendation_system.sql (swipe_events, user_levels, card_queue, etc.)
#
# NOTE: Migration 002 depends on tables from 001 (profiles, flashcards, practice_sessions).
# You must run 001 first if you haven't already.
```

## Running

### Development (with hot reload)

```bash

npm run dev:api
```

## API Endpoints

### Fast Path (called during swiping, <30ms)

| Method | Endpoint | Latency | Description |
|--------|----------|---------|-------------|
| POST | `/api/swipe` | ~10ms | Fire-and-forget swipe recording |
| GET | `/api/cards/next/{userId}` | ~20ms | Pre-loaded cards from queue |

### Queue Management (called on app open)

| Method | Endpoint | Description |
|--------|----------|-------------|
| POST | `/api/queue/prefill` | Ensure queue has cards ready |

### Slow Path (on-demand)

| Method | Endpoint | Description |
|--------|----------|-------------|
| POST | `/api/recommendations` | On-demand AI recommendations |
| POST | `/api/level` | Get/recalculate user CEFR level |
| GET | `/api/interests/{userId}` | Topic interest scores |
| GET | `/api/due-cards/{userId}` | Spaced repetition due cards |
| GET | `/health` | Service health check |

## Architecture

The system is designed so **the user never waits for AI**. Cards are pre-computed in a background queue and served instantly.

```
 ┌────────────────────────────────────────────────────────────┐
 │                    Mobile / Web App                         │
 │                                                             │
 │  1. App opens → POST /api/queue/prefill  (fills queue)      │
 │  2. Show cards → GET /api/cards/next/:id  (instant, ~20ms)  │
 │  3. User swipes → POST /api/swipe   (fire & forget, ~10ms)  │
 │  4. Running low → GET /api/cards/next/:id (refills in bg)   │
 └────────────┬─────────────────────┬─────────────────────────┘
              │ fire & forget       │ instant read
              ▼                     ▼
 ┌────────────────────────────────────────────────────────────┐
 │              Vocally API (port 5000)                        │
 │                                                             │
 │  FAST PATH (sync):              SLOW PATH (background):     │
 │  ┌──────────────────┐           ┌───────────────────────┐   │
 │  │  GET /cards/next  │           │  Background Worker    │   │
 │  │  reads card_queue │           │                       │   │
 │  │  ~15ms, no AI     │           │  • Record swipe       │   │
 │  └──────────────────┘           │  • Update topic score  │   │
 │                                  │  • SM-2 scheduling    │   │
 │                                  │  • Level detection    │   │
 │                                  │  • LangChain AI rank  │   │
 │                                  │  • Refill card_queue  │   │
 │                                  └───────────────────────┘   │
 └────────────────────────────────────────────────────────────┘
              │
              ▼
       ┌─────────────────┐
       │ Supabase (PgSQL) │
       │                   │
       │ card_queue (fast) │
       │ swipe_events      │
       │ user_levels       │
       │ spaced_repetition │
       │ topic_interests   │
       └─────────────────┘
```

## Client Integration

```typescript
const API_URL = 'http://localhost:5000'; // or your deployed URL

// 1. On app open — prefill the queue
await fetch(`${API_URL}/api/queue/prefill`, {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({ user_id: userId }),
});

// 2. Load first batch of cards (instant ~20ms)
const { cards, remaining } = await fetch(
  `${API_URL}/api/cards/next/${userId}?count=10`
).then(r => r.json());

// 3. On each swipe — fire and forget (don't await)
fetch(`${API_URL}/api/swipe`, {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({
    user_id: userId,
    flashcard_id: card.id,
    direction: 'right', // or 'left'
    response_time_ms: swipeDuration,
  }),
});

// 4. When cards are running low — fetch more
if (remaining < 3) {
  const { cards: more } = await fetch(
    `${API_URL}/api/cards/next/${userId}?count=10`
  ).then(r => r.json());
}
```

## How It Works

### 1. Level Detection
Every swipe is recorded with the card's difficulty and IELTS band. The system calculates per-category accuracy rates, weighted accuracy per IELTS band, and maps to CEFR (A1-C2) with a confidence score. Level re-assessment triggers every 20 swipes.

### 2. Topic Interest Tracking
Tracks engagement per category using swipe direction, response time, and a Zone of Proximal Development model (~65% accuracy = most interesting). Uses exponential moving average for smooth scoring.

### 3. Spaced Repetition (SM-2)
Implements the SuperMemo SM-2 algorithm. Response time fine-tunes quality scores (0-5). Right swipe = schedule further out. Left swipe = review tomorrow.

### 4. LangChain AI Ranking
GPT-4o-mini receives the user's profile + candidate cards and ranks them by ZPD fit, topic alignment, thematic coherence, novelty, and practical value.

### 5. Card Queue (the fast part)
All recommendations are pre-computed into a `card_queue` table with denormalized card data. The GET endpoint reads from this table with a single indexed query — no joins, no AI calls. Background workers keep the queue filled (target: 15 cards, refill at 5).
